# PRODUCT_PHILOSOPHY.md

# RTBrowser Product Philosophy

## Vision

RTBrowser is a compact native Windows trading-focused workstation browser optimized for speed, density, responsiveness, and efficient multi-tab workflows.

RTBrowser is designed primarily for TradingView-heavy workflows while remaining lightweight, predictable, and workstation-oriented.

---

# What RTBrowser Is

RTBrowser is:

* A native WinUI browser shell
* A WebView2-powered lightweight workstation browser
* A compact productivity-focused browser
* A trading-oriented browsing environment
* A keyboard-efficient browser
* A portable-first application
* A performance-focused runtime

---

# What RTBrowser Is NOT

RTBrowser is NOT:

* An Arc clone
* A Chromium fork
* An Electron browser
* A social/productivity platform
* An AI-first browser
* An extension ecosystem
* A theme-heavy customizable browser
* A cloud-dependent browser

---

# Core Priorities

Priority order:

1. Responsiveness
2. Compactness
3. Stability
4. Memory efficiency
5. Workflow speed
6. Trading layouts
7. Visual polish

---

# Compactness Philosophy

Every pixel must justify existing.

RTBrowser prioritizes:

* dense layouts
* minimal chrome
* efficient spacing
* compact controls
* low visual noise

The browser UI should remain:

* silent
* predictable
* compact
* workstation-oriented

---

# Performance Philosophy

Performance is a product feature.

RTBrowser targets:

* under 1 second startup
* instant UI responsiveness
* lazy initialization
* lightweight rendering
* aggressive memory management
* minimal animations

RTBrowser avoids:

* unnecessary acrylic
* heavy transparency
* complex animation systems
* HTML-rendered browser chrome

---

# UX Philosophy

RTBrowser should feel:

* fast
* stable
* calm
* dense
* efficient

RTBrowser avoids:

* distracting motion
* playful gimmicks
* floating UI behavior
* excessive onboarding
* visual clutter

---

# Runtime Philosophy

Tabs are lightweight state containers.

WebViews are managed runtime resources.

RTBrowser uses:

* sleeping tabs
* lazy restoration
* WebView pooling
* aggressive unloading under pressure
* lifecycle management

---

# Feature Acceptance Rule

Every feature must answer:

Does this make RTBrowser:

* faster?
* cleaner?
* more focused?
* more efficient?

If not, the feature should be rejected.

---

# Platform Philosophy

RTBrowser is Windows-only.

RTBrowser prioritizes:

* WinUI native rendering
* WebView2 integration
* portable-first deployment
* local-first architecture
* low runtime overhead

---

# Final Goal

RTBrowser should eventually feel like:

"A native trading workstation that happens to browse the web."
